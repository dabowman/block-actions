# CLAUDE.md ‚Äî Block Actions

## Project Overview

**Block Actions** is a WordPress plugin that extends core blocks (`core/button`, `core/group`) with custom JavaScript actions and data attributes. It lets theme and plugin developers add interactivity (carousels, modals, scroll behavior, etc.) to WordPress blocks without building custom blocks or rebuilding the plugin.

As of v2.0.0, the plugin supports two frontend systems:
- **Legacy system** (v1.x): Imperative DOM manipulation via `BaseAction` class and `window.BlockActions` global API.
- **Interactivity API system** (v2.0.0): Declarative stores using WordPress's `@wordpress/interactivity` with server-side directive injection via PHP renderers.

Both systems coexist, controlled by a feature flag (`use_interactivity_api` setting).

- **Version:** 2.0.0
- **Requirements:** WordPress 6.6+, PHP 8.0+, Node.js 20+
- **License:** GPL-2.0-or-later
- **Namespace (PHP):** `Block_Actions`

### Test architecture notes

#### Legacy action tests
- **ES module actions** (`scroll-to-top.js`, `carousel.js`, `test-action.js`): Import `BaseAction` directly; tests import the `init` function and exercise it with DOM fixtures.
- **IIFE/theme actions** (`copy-to-clipboard.js`, `modal-toggle.js`, `smooth-scroll.js`, `toggle-visibility.js`): Use `window.BlockActions` globals; tests set up the global API with real `BaseAction`, then `require()` the action to trigger its IIFE and extract the `init` function from `registerAction` mock calls.
- **`actions/index.js`**: Uses webpack's `require.context` (unavailable in Jest). Tests mock the module to verify whitelist filtering and label generation logic.
- **`block-extensions.js`**: Requires mocking all `@wordpress/*` packages and `lodash`. HOC tests call the filter functions extracted from `addFilter` mock calls and inspect `wp.element.createElement` output.
- **Rate limiting**: `canExecute()` allows 5 executions per second (not 1). Tests must call it 5 times before expecting `false`. Uses `jest.useFakeTimers()` and `jest.advanceTimersByTime()`.

#### Interactivity API store tests (v2.0.0)
- **Mock setup**: `tests/__mocks__/interactivity.js` provides controllable `getContext()` and `getElement()`. Use `__setContext(ctx)` and `__setElement({ ref })` before each test.
- **Store capture**: Tests use `beforeAll` with `require()` to load the store module once, then extract the `storeDefinition` from `store.mock.calls`. Avoid `jest.isolateModules` ‚Äî it creates a separate mock registry and breaks `store.mock.calls` capture.
- **Carousel tests**: `jest.useFakeTimers()` in `beforeEach` overrides the `requestAnimationFrame` mock; re-mock RAF in `beforeEach` after `jest.useFakeTimers()`.
- **Legacy bridge tests**: Use unique action IDs per test to avoid collisions with `store.mock.calls.find()`.
- **Utility tests**: `rate-limiter.test.js`, `sanitize.test.js`, `api.test.js` are straightforward unit tests.

## Architecture

### Dual Frontend Systems

The plugin supports two frontend rendering strategies:

#### Legacy System (v1.x, default)
1. **Editor side** (`block-extensions.js`): Uses WordPress `hooks` to add `data-action` and `data-*` attributes to supported blocks via inspector panel controls.
2. **Frontend side** (`frontend.js`): On `DOMContentLoaded`, finds all `[data-action]` elements, looks up the action in the registry, and calls its `init(element)` function.
3. **Action registry** (`actions/index.js`): Uses webpack `require.context` to auto-discover action files. Only actions in the `CORE_ACTION_IDS` whitelist are shipped as built-in.
4. **Theme actions**: Theme devs drop `.js` files in their theme's `/actions` directory using `window.BlockActions.registerAction()`. PHP auto-discovers and enqueues them.

#### Interactivity API System (v2.0.0, opt-in)
1. **PHP renderers** (`includes/renderers/*.php`): Each action has a renderer that declares initial context and directives. Extends `Action_Renderer` abstract class.
2. **Directive Transformer** (`includes/class-directive-transformer.php`): Hooked to `render_block` filter. Uses `WP_HTML_Tag_Processor` to find `data-action` on root elements and inject `data-wp-interactive`, `data-wp-context`, and action-specific directives.
3. **JS stores** (`src/stores/*/view.js`): Each action is an Interactivity API `store()` with `state`, `actions`, and `callbacks`. Loaded as script modules via `wp_enqueue_script_module()`.
4. **Legacy bridge** (`src/stores/compat/legacy-bridge.js`): Wraps old-style `window.BlockActions.registerAction()` IIFE theme actions into Interactivity API stores automatically.

Enable via: Settings ‚Üí Block Actions ‚Üí "Use Interactivity API"

### How the Interactivity API Pipeline Works

```
Block saved with data-action="carousel"
        ‚Üì
PHP render_block filter fires
        ‚Üì
Directive_Transformer finds data-action, looks up Carousel renderer
        ‚Üì
Carousel renderer injects:
  - data-wp-interactive="block-actions/carousel"
  - data-wp-context='{"currentIndex":0,...}'
  - data-wp-init="callbacks.init"
  - data-wp-on--keydown="actions.handleKeydown"
  - Child directives via post_process_html() (buttons, slides, thumbnails)
        ‚Üì
Carousel view script module enqueued via wp_enqueue_script_module()
        ‚Üì
Browser: Interactivity API processes directives, store hydrates
```

### Key Extension Points

- **`window.BlockActions.registerAction(id, label, init)`** ‚Äî Register actions from theme JS (works in both legacy and Interactivity API modes via legacy bridge)
- **`window.BlockActions.BaseAction`** ‚Äî Base class with XSS protection, rate limiting, telemetry (legacy mode)
- **`block_actions_directories` filter (PHP)** ‚Äî Add custom action directories
- **`block_actions_enable_csp` / `block_actions_csp_header` filters (PHP)** ‚Äî CSP header control
- **`Action_Renderer` abstract class (PHP)** ‚Äî Extend for custom Interactivity API action renderers

### Adding a New Built-in Action

#### For Interactivity API (v2.0.0):
1. Create `src/stores/your-action/view.js` with an Interactivity API `store()` call
2. Create `includes/renderers/class-your-action.php` extending `Action_Renderer`
3. Register the renderer in `block-actions.php` ‚Üí `init_interactivity_api()`
4. Add the webpack entry in `webpack.config.js`
5. Add test files in `tests/stores/`
6. Run `npm run build`

#### For Legacy System:
1. Create `src/actions/your-action.js` with a default export `init(element)` function
2. Add `'your-action'` to `CORE_ACTION_IDS` in `src/actions/index.js`
3. Add a test file at `tests/actions/your-action.test.js`
4. Run `npm run build`

Or use the scaffolding tool: `npm run create-action`

## Code Conventions

### JavaScript
- ES6+ (arrow functions, const/let, template literals, modules)
- No jQuery ‚Äî vanilla JS only
- JSDoc on all exported functions with `@since` tags (1.0.0 for legacy, 2.0.0 for new)
- camelCase for functions/variables, UPPER_SNAKE_CASE for constants, kebab-case for filenames
- JSX uses WordPress pragmas (`wp.element.createElement`), not React
- Spaces for indentation in JS (not tabs, despite WordPress standards)
- Interactivity API stores use `store()` from `@wordpress/interactivity`
- Generator functions (`function*`) for async Interactivity API actions (e.g., copy-to-clipboard)

### PHP
- Namespaced under `Block_Actions` (renderers under `Block_Actions\Renderers`)
- WordPress coding standards: tabs for indentation, snake_case functions
- All output escaped (`esc_html`, `esc_url_raw`, etc.)
- All input sanitized via `sanitize_callback`
- Comprehensive docblocks with `@since`, `@param`, `@return` tags
- Uses `WP_HTML_Tag_Processor` for safe HTML manipulation (not regex)

### Error Handling
- Try/catch in all WordPress filter callbacks; return unmodified state on error
- Centralized logging via `log(type, message, error)` utility in stores
- Legacy: centralized logging via `BaseAction.log(type, message, error)`
- Graceful degradation over hard failures

### Security
- XSS protection via DOMPurify (text) and style whitelist (both systems)
- CSRF via WordPress nonce in API requests
- Rate limiting: WeakMap-based per-element in stores, BaseAction-based in legacy (max 5 executions/second)
- CSP headers available (opt-in via setting or filter)

## Important Notes

- **Supported blocks:** Only `core/button` and `core/group` receive action controls in the editor
- **Build output is committed:** The `build/` directory is included in the repo for distribution
- **Theme actions don't require rebuild:** They're plain JS files auto-discovered by PHP
- **Legacy bridge:** IIFE theme actions using `registerAction()` work in Interactivity API mode via the legacy bridge shim
- **Action IDs come from filenames:** `my-action.js` ‚Üí action ID `my-action`
- Files prefixed with `_` or `.` in theme action directories are skipped
- **No ESLint config:** The project has no `.eslintrc`; `wp-scripts lint-js` uses defaults
- **Feature flag:** The `use_interactivity_api` setting controls which frontend system is active. Both can coexist.
- **Script modules:** Interactivity API stores are enqueued via `wp_enqueue_script_module()` with `@wordpress/interactivity` as a dependency
